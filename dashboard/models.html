Model Validation
Two trained, cross-validated models — not hand-tuned rules
Why These Two Targets

We first tried training on priority — and found it's almost perfectly determined by corridor alone: every named corridor is ~100% "High", "Non-corridor" is ~99.8% "Low". It's an operational designation in the source system, not a judgment call, and a model "predicting" it would report a meaningless 99.9% AUC — measuring a database join, not a prediction.

So this project doesn't train on priority at all. Instead it models the two signals in this dataset that carry genuine, non-trivial uncertainty: whether an event will require a road closure (varies continuously from 4.3% for breakdowns to 80% for VIP movement, with no deterministic rule), and how long an event will take to resolve — honestly the noisiest signal here, which is why the dashboard always shows it as a range, not a single number.
